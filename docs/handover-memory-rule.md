# 핸드오버 메모리 작성 규칙

> 세션 경계(페이즈 완료·도메인 전환·컨텍스트 리셋 직전)에 다음 세션이 즉시 복귀할 수 있도록
> `memory/project_*_complete.md` 또는 `memory/session_handover_*.md` 파일을 작성하는 규칙.
>
> **이 문서는 git 저장소에 영구 보관**되며, Claude Desktop 재설치·설정 초기화 이후에도
> `docs/handover-memory-rule.md` 경로로 접근 가능하다.

---

## 1. 자동 트리거 (명시적 요청 없이 실행)

| 트리거 | 메모리 파일명 예시 |
|--------|------------------|
| 페이즈·RP·도메인 단위 완료 커밋 + 푸시 직후 | `project_pixel_port_rp5_handover.md` |
| 세션 리셋·`/clear` 직전 (세션 핸드오버 규칙과 연동) | `session_handover_2026_04_21.md` |
| Compaction 3회 이상 누적 + 클린 커밋 경계 도달 | `project_session{N}_complete.md` |

**예외:** 사용자가 "이번은 handover 메모리 건너뛰어"를 명시하면 스킵.

---

## 2. 7섹션 필수 구조

모든 핸드오버 메모리는 아래 7개 섹션을 순서대로 포함해야 한다.
섹션을 생략하거나 순서를 변경하면 품질 체크리스트 탈락.

### 섹션 1 — frontmatter (완료 상태 헤더)

```markdown
---
name: {작업단위} 완료 → {다음 작업단위} 인수인계
description: {브랜치} 브랜치 {작업단위} 완료 시점 상태 + 다음 진입 컨텍스트
type: project
---

# {작업단위} 완료 상태 ({YYYY-MM-DD})

**브랜치:** `{branch-name}`
**마지막 커밋:** `{short-hash} {커밋 메시지 한 줄}`
**원격:** 푸시 완료 / 미푸시
**추천 모델:** `{claude-sonnet-4-6 | claude-opus-4-7}` — {선택 이유 한 줄}
```

### 섹션 2 — 완료 내용

- 커밋 해시별 요약 (각 커밋이 무엇을 했는지 한 줄)
- 신규/변경된 핵심 파일 목록
- canonical 데이터 소스 (타입·상수·헬퍼가 정의된 파일)

### 섹션 3 — Deferred

- 처리하지 못하고 다음으로 넘긴 항목 목록 (D1~Dk)
- 각 항목에 **왜 defer 했는지** + **언제 처리할 것인지** 명시
- `docs/code-review-deferred.md` 해당 섹션과 교차 링크

### 섹션 4 — 결과 문서 포인터 (이양 가능 링크)

> **핵심 원칙:** handover 메모리 하나만 읽으면 다음 세션 Claude가 모든 관련 결과 문서 위치를 알 수 있어야 한다.
> "이양 가능한 결과 문서"란 곧 "이 섹션의 포인터 목록에서 출발해 단 한 번의 Read로 도달 가능한 문서"를 의미한다.
> 경로를 추측하게 만들지 말 것.

포함 필수:
- `docs/milestone.md` 의 해당 세션/페이즈 섹션 (섹션명 또는 line 번호)
- `docs/code-review-deferred.md` 의 해당 섹션 (존재할 경우)
- 이번 세션에서 생성·갱신된 모든 결과 문서 경로
- 이전 핸드오버 메모리 (체인 유지 — N-1 → N → N+1 추적 가능하도록)

### 섹션 5 — 진입 컨텍스트

- 다음 세션 작업 대상 + 목적
- 참조해야 할 프로토타입 위치 또는 디자인 문서 경로
- 재사용 가능한 기존 자산 목록
- 새로 작성해야 하는 파일·컴포넌트 목록
- 주의사항·예외 케이스 (결정된 정책·확정된 제약)

### 섹션 6 — 영구 원칙

- 이 작업 도메인에서 반드시 준수할 feedback 메모리 링크 목록
- 포맷: `- **{원칙명}:** ({memory/feedback_*.md})`
- 새 세션의 Claude가 스킵하지 않도록 2~5건으로 압축

### 섹션 7 — 다음 세션 첫 단계

- 번호가 붙은 순서대로 실행할 초기 액션 (3~5단계)
- 첫 단계는 항상 상태 검증 (git log 또는 마일스톤 확인)
- 마지막 단계는 "사용자 승인 후 착수"로 끝내기 (자동 착수 방지)

## 3. 파일명 컨벤션

| 상황 | 파일명 패턴 |
|------|-----------|
| pixel-port RP-N 완료 | `project_pixel_port_rp{N}_handover.md` |
| 세션 단위 완료 | `project_session{N}_complete.md` |
| 날짜 기반 세션 핸드오버 | `session_handover_{YYYY_MM_DD}.md` |
| 도메인 전환 (백→프론트 등) | `project_{domain}_{topic}_entry.md` |

파일은 `~/.claude/projects/{project}/memory/` 에 저장한다.
동시에 `MEMORY.md` `## Project` 섹션에 한 줄 인덱스 추가 필수.

## 5. 품질 체크리스트

핸드오버 메모리 작성 후 제출 전 아래를 자가 검증한다.

- [ ] 7섹션 모두 존재 (frontmatter·완료 내용·Deferred·결과 포인터·진입 컨텍스트·영구 원칙·다음 첫 단계)
- [ ] 섹션 4 결과 문서 포인터: 각 경로가 실제 파일로 존재하는지 확인 (추측 경로 금지)
- [ ] 섹션 4에 이전 핸드오버 메모리 체인 링크 포함
- [ ] Deferred 항목마다 "왜 defer 했는가" + "언제 처리할 것인가" 명시
- [ ] 커밋 해시가 최소 하나 이상 포함 (완료 시점 기록)
- [ ] MEMORY.md `## Project` 섹션에 한 줄 인덱스 추가 완료
- [ ] 파일명이 컨벤션 패턴과 일치

---

## 6. 안티 패턴

| 금지 패턴 | 대안 |
|----------|------|
| 결과 문서 경로 추측 기재 (존재 확인 안 함) | Write 전 Glob/Grep 으로 경로 검증 |
| Deferred 항목에 이유 없이 "나중에 처리" | "왜 defer + 언제 처리" 명시 |
| 이전 핸드오버 체인 누락 | 섹션 4에 이전 파일 링크 필수 |
| 너무 긴 서사형 설명 | 섹션별 핵심만, 필요 시 원본 문서 링크 |
| 커밋 없이 handover 메모리 작성 | 먼저 커밋·푸시, 해시 확인 후 작성 |
| MEMORY.md 인덱스 누락 | 파일 작성과 동시에 인덱스 추가 |
